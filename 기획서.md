# Drift — 프로젝트 의존성 건강 모니터

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **이름** | Drift |
| **한줄 소개** | 프로젝트 의존성의 "생존 확률"을 점수화하고 죽어가는 라이브러리를 사전 경고하는 CLI |
| **카테고리** | Developer Tooling / Dependency Management |
| **라이선스** | MIT |
| **난이도** | 중 |
| **예상 개발 기간** | MVP 4~6주 |

## 2. 문제 정의

### 현재 상황
- 오픈소스 메인테이너 **60%**가 번아웃 경험 (Tidelift 2024 Survey)
- 매년 수천 개의 npm 패키지가 방치됨 — `event-stream` 사태(2018), `colors.js` 사태(2022) 반복
- `npm audit`은 **보안 취약점만** 체크하고, 라이브러리의 **건강 상태**는 무시
- 개발자는 "이 라이브러리가 6개월 뒤에도 살아있을까?"라는 질문에 답할 수 없음

### 핵심 고통점
1. **사후 대응**: 라이브러리가 죽은 후에야 알게 됨
2. **건강 지표 분산**: GitHub 마지막 커밋, npm 다운로드 추세, 이슈 응답율 등을 각각 확인해야 함
3. **대안 부재**: `npm audit` ≠ 건강 체크. OSSF Scorecard는 보안 중심 + UX 부족
4. **기업 리스크**: 핵심 의존성이 죽으면 마이그레이션 비용 막대

### 학술 연구 동향
- **ICSE 2026**: "Abandabot" 논문 발표 — 같은 컨셉의 학술 연구가 시작됨
- 실용적 CLI 도구는 아직 없음 → **선점 기회**

### 건강 점수 모델 (0~100)

| 신호 | 가중치 | 측정 방식 |
|------|-------|----------|
| **마지막 커밋** | 20% | 최근 커밋 날짜 (3개월 이내=만점) |
| **릴리스 주기** | 15% | 릴리스 간격의 규칙성과 빈도 |
| **메인테이너 수** | 15% | 활성 커미터 수 (bus factor) |
| **이슈 응답율** | 15% | 이슈 오픈→첫 응답 시간 |
| **다운로드 추세** | 15% | 최근 6개월 다운로드 증감 추세 |
| **CVE 이력** | 10% | 보안 취약점 발생 빈도 + 패치 속도 |
| **커뮤니티 신호** | 10% | Star 추세, Fork 활성도, PR 머지율 |

### 작동 방식
```bash
drift check                    # 현재 프로젝트 전체 스캔
drift check axios lodash       # 특정 패키지만 체크
drift watch                    # CI에서 지속 모니터링 (점수 하락 시 알림)
drift suggest moment           # 대안 라이브러리 추천
drift report --format json     # JSON 출력 (CI 통합용)
drift compare axios vs ky      # 두 라이브러리 건강 비교
```

## 5. 핵심 기능

### MVP (v0.1)
- [ ] `drift check` — package.json 분석 + 건강 점수 계산
- [ ] GitHub API 기반 메인테이너 수, 마지막 커밋, 이슈 응답율
- [ ] npm API 기반 다운로드 추세
- [ ] 터미널 컬러 출력 (점수별 색상)

- [ ] 캐싱 시스템 (API 호출 최소화)
- [ ] pip (Python), cargo (Rust), go.mod 지원

- [ ] 건강 추세 그래프 (시간별 변화)

### v1.0

## 6. 기술 스택

| 구성 요소 | 기술 | 선택 이유 |
|----------|------|----------|
| **언어** | Rust | 빠른 실행 속도, 크로스 플랫폼 바이너리 |
| **HTTP 클라이언트** | reqwest | Rust 표준 HTTP 라이브러리 |
| **CLI 프레임워크** | clap | Rust CLI 표준 |
| **출력 포맷** | comfy-table + colored | 터미널 테이블 + 색상 |
| **캐싱** | SQLite (rusqlite) | 로컬 캐시 DB |
| **패키지 파서** | serde_json / serde_yaml | package.json, Cargo.toml 등 파싱 |
| **API** | GitHub REST API, npm Registry API | 데이터 소스 |

### 대안: TypeScript 구현
npm 생태계 접근성을 위해 TypeScript로 구현하는 것도 고려 가능.
→ `npx drift check`로 즉시 실행 가능한 이점

## 7. 경쟁 분석

| 프로젝트 | 상태 | 차이점 |
|---------|------|--------|
| OSSF Scorecard (4.5k⭐) | 활발 | 보안 중심, UX 부족, 프로젝트 단위 (의존성 아님) |
| socket.dev | 상용 | 보안 중심 SaaS, OSS 아님 |
| Snyk | 상용 | 취약점 스캐너, 건강 점수 아님 |
| npm audit (내장) | 활발 | 보안 취약점만, 방치/건강 무시 |
| Abandabot (학술) | 논문 | CLI 도구 없음, 학술 연구 단계 |
| deps.dev (Google) | 활발 | 웹 전용, CLI 없음, 점수 방식 다름 |

### Drift의 차별점
1. **건강 중심**: 보안이 아닌 "생존 가능성"에 집중
2. **예측적**: 현재 상태뿐 아니라 미래 예측
3. **Actionable**: 문제만 보여주는 게 아니라 대안까지 제시
4. **다중 생태계**: npm, pip, cargo, go 모두 지원

## 8. 구현 계획

### Phase 1: 기반 (1~2주)
- CLI 프레임워크 구축 (check 명령어)
- package.json 파서
- GitHub API 연동 (마지막 커밋, 메인테이너 수)
- npm Registry API 연동 (다운로드 추세)

### Phase 2: 점수 엔진 (2~3주)
- 건강 점수 계산 모델 구현
- 이슈 응답율 분석
- CVE 이력 조회 (OSV API)
- 터미널 출력 포맷팅

### Phase 3: 고급 기능 (3~5주)
- 캐싱 시스템 (API rate limit 대응)
- `drift suggest` 대안 추천 엔진
- `drift watch` CI 모드
- JSON/SARIF 출력 지원

### Phase 4: 출시 (5~6주)
- 크로스 플랫폼 빌드 (GitHub Actions)
- npm/cargo/homebrew 배포
- README + 데모 GIF
- Hacker News / Reddit 런칭

## 10. 리스크 및 대응

| 리스크 | 확률 | 대응 |
|--------|-----|------|
| GitHub API rate limit | 높음 | 캐싱 + conditional requests + GraphQL 전환 |
| 건강 점수 정확도 논란 | 높음 | 투명한 점수 산정 공식 공개, 커뮤니티 피드백 반영 |
| Abandabot 논문이 도구로 발전 | 중간 | 빠른 선점 + 커뮤니티 구축으로 방어 |
| npm audit이 건강 점수 도입 | 낮음 | 다중 생태계 + 예측 기능으로 차별화 유지 |
